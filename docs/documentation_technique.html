Présentation du projet

HealthAI MSPR est une application web full-stack de suivi et d'analyse de données de santé, conçue pour EPSI. Elle permet la collecte, la visualisation et l'analyse des données physiologiques des utilisateurs (activité physique, alimentation, métriques santé, objectifs).

Objectifs

  • ✓Centraliser les données de santé des utilisateurs
  • ✓Offrir un tableau de bord d'analyse pour les administrateurs
  • ✓Permettre aux utilisateurs de consulter leur profil santé
  • ✓Gérer les flux de données et leur validation
  • ✓Détecter et nettoyer les données incohérentes
  • ✓Automatiser les pipelines de données avec Airflow
  • ✓Sécuriser l'accès via authentification JWT
  • ✓Déploiement 100% conteneurisé avec Docker

Architecture technique

L'application repose sur une architecture full-stack conteneurisée et découplée. Le frontend Vue 3 est servi par Nginx, qui joue aussi le rôle de reverse proxy vers l'API FastAPI. Le backend applique les règles métier, sécurise les accès par JWT et persiste les données dans PostgreSQL Neon. En parallèle, Airflow orchestre les traitements de données, tandis que Prometheus, Grafana, Loki et Promtail assurent le monitoring et la centralisation des logs.

Réseau Docker

Les conteneurs applicatifs communiquent via le réseau bridge interne health_net. La pile monitoring utilise le même réseau Docker pour atteindre backend, frontend, prometheus et loki. La base applicative n'est pas conteneurisée : elle est fournie par Neon et jointe uniquement via DATABASE_URL. Seuls les ports nécessaires sont exposés vers l'hôte.

API REST — Endpoints

L'API est documentée automatiquement via Swagger UI à l'adresse http://localhost:89/api/docs (accès public via Nginx) et proxifiée par Nginx sur le port 89. Les routes publiques sont /api/login, /api/utilisateurs/register et /api/health. Les routes métier requièrent un token JWT Bearer, et /api/metrics est réservé aux administrateurs. Toutes les routes sont exposées publiquement sous le préfixe /api via API_ROOT_PATH et le reverse proxy Nginx du VPS.

Structure du token JWT

Header

{ "alg": "HS256", "typ": "JWT" }

Payload

{ "sub": "1001", // ID utilisateur "is_admin": false, // Rôle "exp": 1741945200 // +8 heures }

Signature

HMACSHA256( base64(header) + base64(payload), SECRET_KEY // .env )

Paramètres de sécurité

Paramètre Valeur Rôle
ALGORITHM HS256 Algorithme de signature symétrique
ACCESS_TOKEN_EXPIRE_MINUTES 480 (8h) Durée de validité du token
SECRET_KEY Variable d'environnement Clé secrète de signature (non exposée)
pwd_context bcrypt (rounds auto) Hash adaptatif des mots de passe

Gestion côté frontend

  • Le token est décodé côté client pour extraire sub (user ID) et is_admin
  • isAuthenticated() vérifie à la fois la présence du token ET son expiration (champ exp)
  • Toute réponse 401 provoque un logout automatique et une redirection vers /connexion
  • Le token est stocké dans localStorage sous la clé app_token
  • Les guards de navigation Vue Router bloquent l'accès aux routes protégées si non authentifié

Déploiement production (Docker)

Le frontend est servi en production uniquement par le conteneur frontend_nginx. Aucun serveur de développement Vite ne tourne dans Docker.

Apache Airflow 2.10 orchestre les pipelines de données asynchrones. Interface web sur le port 8080 (identifiants : airflow / airflow). Le backend accède à l'API Airflow via host.docker.internal:8080.

Attribut Valeur
Fichier airflow/dags/incorporation_ml.py

Flux de validation

Les DAGs produisent des CSV horodatés. L'admin visualise, modifie et valide via la page Gestion des flux (onglets Import/Export). Seuls les CSV validés sont incorporés par l'DAG incorporation_ml.

Notes de configuration récentes

  • La base applicative n'est pas un conteneur Docker : DATABASE_URL pointe vers Neon PostgreSQL.

  • Les logs fichier du backend sont persistés dans ./data/logs/backend via le montage /app/logs.

Sécurité

Authentification & Autorisation

  • JWT HS256 — Tokens signés, expiration 8h
  • bcrypt — Hash adaptatif des mots de passe, jamais stockés en clair
  • OAuth2 Password Flow — Standard RFC 6749
  • RBAC — 2 rôles : user et admin
  • Vérification du token à chaque endpoint protégé
  • Déconnexion automatique si token expiré (frontend)

Protection des endpoints

  • Routes publiques : /api/login, /api/utilisateurs/register
  • Routes auth : lecture des données (GET)
  • Routes admin-only : suppressions (DELETE) et modifications globales
  • Endpoint /api/metrics protégé par le rôle administrateur
  • Validation stricte avec Pydantic v2 sur toutes les entrées
  • Path parameters validés (gt=0 pour les IDs)

Infrastructure

  • Ports DB exposés uniquement sur 127.0.0.1
  • SECRET_KEY via variable d'environnement (jamais en dur)
  • DATABASE_URL Neon et paramètres SMTP chargés depuis .env
  • Réseau Docker interne isolé (health_net)
  • CORS configuré (à restreindre en production)

Frontend

  • Guards de navigation empêchent l'accès non autorisé
  • Vérification expiration JWT côté client (champ exp)
  • Gestion des 401 : logout + redirection automatique
  • Pas de données sensibles exposées dans le DOM

Observabilité backend

/api/metrics Métriques en mémoire : requêtes, erreurs 5xx, durée moyenne, routes appelées. Accès admin requis.

Les jobs sonarqube et deploy s'exécutent en parallèle après les tests, uniquement sur un push vers main.

Conséquence importante : le déploiement attend les tests backend et frontend, mais il n'attend pas le résultat de SonarQube. Un échec du scan ou de la Quality Gate Sonar n'empêche donc pas actuellement la mise en production.

Observabilité production

La stack monitoring est séparée du pipeline CI/CD : elle n'est pas déployée par scripts/deploy.sh ni par GitHub Actions. Elle tourne en parallèle sur le serveur via docker-compose.monitoring.yml.

Architecture

Serveur VPS (/opt/HEALTHAIMSRP)
  ├── Stack app : backend (8089) + frontend (89)
  ├── Stack monitoring : Grafana, Prometheus, Loki, Promtail, exporters
  ├── Nginx VPS : HTTPS public, route /api/* → backend
  └── Neon PostgreSQL (métriques via console Neon)

Tous les ports monitoring sont liés à 127.0.0.1 : accès local ou tunnel SSH, sauf exposition nginx dédiée.

Composants et rôles

Service Port local Rôle
healthaim_grafana 3000 Dashboards — datasources Prometheus et Loki provisionnées
healthaim_prometheus 9090 Collecte métriques (node-exporter, blackbox, self-scrape)
healthaim_loki 3100 Stockage logs (rétention 7 jours)
healthaim_promtail — Envoie data/logs/backend/*.log et logs Docker vers Loki
healthaim_node_exporter 9100 CPU, RAM, disque du VPS
healthaim_blackbox 9115 Sondes HTTP : /api/health, frontend, préfixe /api

Accès Grafana

Élément Valeur
Utilisateur admin
Mot de passe GRAFANA_ADMIN_PASSWORD dans .env
Tunnel SSH ssh -L 3000:127.0.0.1:3000 deploy@serveur → http://localhost:3000
Reset mot de passe ./scripts/reset-grafana-password.sh
GRAFANA_ADMIN_PASSWORD ne s'applique qu'au premier démarrage Grafana. Si le mot de passe ne fonctionne plus, utiliser le script de reset (volume grafana_data déjà initialisé).

Surveillance des routes /api et erreurs 4xx/5xx

Deux mécanismes complémentaires : Loki pour les logs détaillés, Prometheus pour les métriques d'infrastructure et la disponibilité HTTP.

Logs backend → Loki (déjà en place)
