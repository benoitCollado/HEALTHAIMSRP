1. Resume du projet

HealthAI MSPR est une application de suivi sante et sportif. Elle permet de gerer des utilisateurs, aliments, exercices, consommations, activites, metriques sante, objectifs et interactions avec un assistant IA.

Le projet est compose d'un frontend Vue, d'un backend FastAPI, d'un microservice IA, d'un stockage PostgreSQL Neon, d'un stockage objet MiniIO, d'un cache Redis et de pipelines Airflow.

2. Architecture generale

Utilisateur
  -> Frontend Vue / Nginx
    -> Backend FastAPI (/api)
      -> PostgreSQL Neon : donnees relationnelles
      -> Redis : rate limiting, cache et donnees temporaires
      -> MiniIO : images du Chat IA
      -> Mistral : conversation IA
      -> microservice_ia : recommandations calories/exercices
      -> microservice photo : analyse d'image
      -> Airflow : flux admin et pipelines data
      -> Monitoring : metrics, logs, alertes

Frontend

Application Vue avec authentification, pages utilisateur, admin, tableau de bord et Chat IA.

Backend

API FastAPI securisee par JWT. Elle expose les routes metier, orchestre les appels IA et gere la resilience.

Microservices IA

Services separes pour recommandations sport/sante et analyse d'image. Le frontend ne les appelle pas directement.

4. Backend FastAPI

Le backend principal se trouve dans backend/app. Il est organise par responsabilites simples : modeles SQLAlchemy, schemas Pydantic, routeurs FastAPI, dependances, securite, cache et clients externes.

Fichier / dossier Responsabilite
app/main.py Creation FastAPI, middlewares, CORS, routes, login, health, metrics.
app/models Tables SQLAlchemy : utilisateur, aliment, exercice, activite, etc.
app/schemas Validation Pydantic des payloads et reponses API.
app/routers Routes REST par domaine fonctionnel.
app/dependencies.py Session DB, utilisateur courant, garde admin.
app/security.py Hash password, JWT, verification token, TOTP 2FA.
app/external_clients.py Clients HTTP resilients vers Mistral, Airflow et microservices IA.
app/cache.py Cache Redis JSON, TTL, cache stale, cles stables.
app/rate_limit.py Middleware de limitation de charge Redis.
app/object_storage.py Validation, optimisation, upload et URL presignee MiniIO.

Cycle de developpement d'une route

  1. Creer ou modifier le modele SQLAlchemy si la base change.
  2. Ajouter une migration Alembic.
  3. Ajouter les schemas Pydantic.
  4. Creer la route dans le routeur du domaine.
  5. Proteger la route avec get_current_user ou require_admin.
  6. Ajouter un test API.

5. Donnees et stockage

PostgreSQL Neon

PostgreSQL Neon est la base relationnelle principale. Elle contient les donnees metier structurees : utilisateurs, aliments, exercices, consommations, activites, metriques et objectifs.

Table Role
utilisateursComptes, profil, role admin, objectifs personnels, 2FA.
alimentCatalogue alimentaire.
exerciceCatalogue sportif.
consommationAliments consommes par utilisateur.
activiteActivites sportives et calories depensees.
metrique_santePoids, sommeil, pas, frequence cardiaque.
objectifObjectifs utilisateur et suivi.

MongoDB

MongoDB est utilise par microservice_ia lorsque MONGODB_URI est configure. Sans cette variable, le microservice utilise des repositories in-memory pour les tests et le developpement local.

Collection Contenu
usersProfil IA, contraintes, equipements, blessures.
workout_plansProgrammes d'entrainement, seances, calories recommandees.
session_logsFeedbacks, RPE, exercices valides, progression.

MiniIO

MiniIO stocke les images jointes au Chat IA. Les images sont redimensionnees a 640 x 640 maximum, converties en JPEG optimise, puis stockees sous un prefixe par utilisateur. 

users/{id_utilisateur}/chat/{uuid}.jpg

6. Chat IA et microservices

Le Chat IA est expose par le backend principal. Le navigateur ne contacte jamais directement Mistral ou les microservices internes.

Route Role
POST /api/chat/ Conversation HealthAI avec Mistral, historique court et cache Redis.
POST /api/chat/images Upload d'une image vers MiniIO pour l'utilisateur connecte.
POST /api/chat/recommendations Lecture du profil SQL et appel a microservice_ia pour calories et exercices.
POST /api/chat/images/analyze Generation d'une URL MiniIO presignee et appel au microservice photo.

Flux recommandations

Frontend
  -> POST /api/chat/recommendations
    -> Backend lit utilisateurs dans PostgreSQL
    -> Backend appelle MICROSERVICE_IA_URL/api/recommandation_calorique
    -> Backend appelle MICROSERVICE_IA_URL/api/recommandation_exercice
    -> Backend retourne answer + recommendation

Flux analyse photo

Frontend
  -> POST /api/chat/images
    -> Backend stocke l'image dans MiniIO
  -> POST /api/chat/images/analyze
    -> Backend verifie users/{id}/chat/
    -> Backend genere une URL presignee interne MinIO
    -> Backend appelle PHOTO_PROCESSING_API_URL
    -> Backend retourne answer + analysis

La configuration photo est volontairement simple : PHOTO_PROCESSING_API_URL contient l'URL complete du endpoint d'analyse, par exemple http://microservice_photo:8000/analyze. Pour cet appel interne, le backend utilise une URL MinIO signee avec l'endpoint Docker minio:9000. Les URLs renvoyees au navigateur restent publiques via MINIO_PUBLIC_ENDPOINT.

7. Redis, cache et gestion de charge

Rate limiting

Redis limite les rafales de requetes. Les limites sont adaptees selon la route : globale, login et chat. 

RATE_LIMIT_ENABLED=true
RATE_LIMIT_DEFAULT_LIMIT=120
RATE_LIMIT_DEFAULT_WINDOW_SECONDS=60
RATE_LIMIT_LOGIN_LIMIT=10
RATE_LIMIT_CHAT_LIMIT=20

Cache

Cache TTL Usage
Chat IA CHAT_CACHE_TTL_SECONDS Eviter de rappeler Mistral pour une requete identique.
MiniIO MINIO_IMAGE_CACHE_TTL_SECONDS, maximum 180 s Eviter de regenerer souvent les URLs presignees.
Airflow frais AIRFLOW_CACHE_TTL_SECONDS Afficher rapidement les derniers runs.
Airflow stale AIRFLOW_STALE_CACHE_TTL_SECONDS Continuer a afficher la page admin si Airflow tombe.

8. Fallback et continuite de service

Les appels externes passent par un client resilient avec timeout, retries et circuit breaker. Le but est d'eviter qu'une dependance externe bloque tout le backend.

Dependance Comportement en panne
Mistral Retour d'une reponse de secours si le fournisseur est indisponible.
Airflow Retour du cache frais, puis stale, puis etat degrade.
Redis Fail-open pour le rate limiting ; cache ignore si Redis tombe.
MiniIO Erreur controlee seulement sur les routes d'image.
microservice_ia /api/chat/recommendations retourne 503 ou 502.
microservice photo /api/chat/images/analyze retourne 503 ou 502.

9. Securite

  • Authentification par JWT Bearer.
  • Hash des mots de passe avec passlib/bcrypt.
  • Routes admin protegees par require_admin.
  • 2FA TOTP disponible pour renforcer la connexion.
  • CORS configure via CORS_ALLOWED_ORIGINS.
  • Headers HTTP de securite via middleware.
  • Controle du proprietaire des images par prefixe users/{id}/chat/.
  • Secrets fournis par .env, jamais par le frontend.

13. Tests et verification

Commande But
python -m compileall app Verifier que le backend Python compile.
python -m pytest app\tests --no-cov Lancer les tests backend sans contrainte coverage.
npm run lint Verifier le typage frontend.
npm test Lancer les tests frontend.

Les tests backend utilisent SQLite en memoire pour eviter de toucher Neon. Les appels externes sont remplaces par des mocks lorsque c'est necessaire.
