1. Resume du projet

HealthAI MSPR est une application de suivi sante et sportif. Elle permet de gerer des utilisateurs, aliments, exercices, consommations, activites, metriques sante, objectifs et interactions avec un assistant IA.

Le projet est compose d'un frontend Vue, d'un backend FastAPI, d'un microservice IA, d'un stockage PostgreSQL Neon, d'un stockage objet MiniIO, d'un cache Redis et de pipelines Airflow.

2. Architecture generale

Utilisateur
  -> Frontend Vue / Nginx (Markdown assistant via marked + DOMPurify)
    -> Backend FastAPI (/api)
      -> PostgreSQL Neon : donnees relationnelles + objectifs utilisateur
      -> Redis : rate limiting, cache et donnees temporaires
      -> MiniIO : images du Chat IA
      -> LangGraph + Mistral (9 tools) : agent conversationnel
      -> microservice_ia : calories, exercices, programmes v1, MongoDB
      -> microservice photo : analyse d'image
      -> Airflow : flux admin et pipelines data
      -> Monitoring : metrics, logs, alertes

Frontend

Application Vue avec authentification, pages utilisateur, admin, tableau de bord et Chat IA (reponses assistant en Markdown).

Backend

API FastAPI securisee par JWT. Elle expose les routes metier, orchestre les appels IA et gere la resilience.

Microservices IA

microservice_ia : calories, exercices ML, programmes v1, MongoDB. microservice_photo : analyse d'image. Non appeles directement par le frontend.

Architecture technique detaillee

L'architecture est separee en couches. Le navigateur appelle uniquement les routes publiques du backend sous /api. Le backend orchestre ensuite les acces SQL, cache, stockage objet, IA externe et microservices internes sur le reseau Docker.

Couche Composants Responsabilite Communication
Presentation Frontend, Nginx Interface Vue, Chat IA, pages utilisateur/admin, rendu Markdown securise. HTTP public vers /api/*.
API principale backend_api, FastAPI JWT, RBAC, CRUD, Chat IA, orchestration microservices, fallback. Expose /api via Nginx ; routes internes FastAPI sans prefixe proxy.
Donnees relationnelles PostgreSQL Neon / PostgreSQL Docker local Utilisateurs, objectifs, aliments, exercices, consommations, activites, metriques. SQLAlchemy via DATABASE_URL.
Cache et charge redis Rate limiting, cache Chat IA, cache Airflow, cache URLs MinIO. redis://redis:6379/0 depuis Docker.
Stockage objet minio Images Chat IA sous users/{id}/chat/{uuid}.jpg. MINIO_ENDPOINT=minio:9000 interne ; MINIO_PUBLIC_ENDPOINT pour navigateur.
IA conversationnelle LangGraph + Mistral tools Orchestration du Chat IA, etat de conversation, selection et execution de tools sante/sport. Backend uniquement, cle API jamais exposee au frontend.
Microservice recommandations microservice_ia Calories, exercices, programmes v1, ML Random Forest, MongoDB optionnel. http://microservice_ia:8090/api/* depuis le backend.
Microservice photo microservice_photo Analyse des images jointes au Chat IA. http://microservice_photo:8000/analyze avec URL MinIO interne signee.
Data pipeline Airflow Imports, exports, nettoyage, flux admin. Backend vers API Airflow, cache frais/stale Redis.

Orchestration LangGraph pour le Chat IA

Le Chat IA utilise LangGraph cote backend pour organiser le raisonnement de l'assistant sous forme de graphe. Cette couche evite d'avoir un simple appel direct au modele : elle permet de gerer l'etat de la conversation, de choisir les bons outils et de recuperer les donnees utiles avant de construire la reponse finale.

Etape Role dans le Chat IA
Message utilisateur Le backend recoit la question depuis /api/chat/ et prepare le contexte utilisateur.
Graphe LangGraph LangGraph decide si la reponse peut etre generee directement ou si un tool doit etre appele.
Tools sante Les tools peuvent lire les donnees SQL, utiliser le cache Redis ou appeler les microservices IA.
Microservices IA Les recommandations passent par microservice_ia avec les routes /api/recommandation_*.
Reponse finale Le backend renvoie une reponse Markdown au frontend, nettoyee ensuite cote interface.

Regles de routage

  • Les routes appelees par le navigateur commencent par /api.
  • Les routes FastAPI internes gardent leur prefixe applicatif, par exemple /chat.
  • Le backend appelle le microservice IA avec /api/recommandation_*.
  • Le backend appelle le microservice photo avec /analyze.
  • Les images utilisent une URL MinIO publique pour le navigateur et une URL MinIO interne pour les microservices Docker.

Cycle de developpement d'une route

  1. Creer ou modifier le modele SQLAlchemy si la base change.
  2. Ajouter une migration Alembic.
  3. Ajouter les schemas Pydantic.
  4. Creer la route dans le routeur du domaine.
  5. Proteger la route avec get_current_user ou require_admin.
  6. Ajouter un test API.

6. Chat IA et microservices

Le Chat IA est expose par le backend principal (Frontend/src/pages/ChatAssistant.vue). Le navigateur ne contacte jamais directement Mistral ni les microservices internes. Les reponses de l'assistant sont rendues en Markdown (Frontend/src/utils/renderMarkdown.ts : marked + DOMPurify). Les messages utilisateur restent en texte brut.

Routes backend chat

Route Role
POST /api/chat/ Conversation via agent LangGraph (chat_graph.py) : profil SQL + objectifs PostgreSQL, Mistral avec 9 tools, cache Redis. Reponse : answer + recommendation optionnel.
POST /api/chat/images Upload d'une image vers MiniIO pour l'utilisateur connecte.
POST /api/chat/recommendations Appel direct microservice (calories + exercices), sans LangGraph.
POST /api/chat/images/analyze URL MiniIO presignee + appel microservice photo.

Agent LangGraph (backend)

POST /api/chat/
  -> chat.py : profil Utilisateur + table objectif (PostgreSQL)
  -> chat_graph.py :
       enrich_context (objectifs normalises pour le microservice)
       detect_requirements (intent : calories, exercices, programme…)
       invoke_model (Mistral + tool_choice force si tool requis)
       run_tools (execute_microservice_tool)
       finalize (reponse basee sur les resultats tools)
  -> Mistral reformule ; pas d'invention de chiffres sans tool

Flux conversation principale

Frontend ChatAssistant.vue
  -> POST /api/chat/ { message, history }
    -> Backend : objectifs + profil SQL
    -> LangGraph + Mistral (tools)
    -> microservice_ia (selon tool appele)
    -> answer (Markdown) + recommendation JSON
  -> Frontend : renderMarkdown(answer) pour l'assistant

Flux recommandations directes

Frontend
  -> POST /api/chat/recommendations
    -> Backend lit utilisateurs dans PostgreSQL
    -> POST MICROSERVICE_IA_URL/api/recommandation_calorique
    -> POST MICROSERVICE_IA_URL/api/recommandation_exercice
    -> Backend retourne answer + recommendation

Flux analyse photo

Frontend
  -> POST /api/chat/images
    -> Backend stocke l'image dans MiniIO
  -> POST /api/chat/images/analyze
    -> Backend verifie users/{id}/chat/
    -> Backend genere une URL presignee interne MinIO
    -> Backend appelle PHOTO_PROCESSING_API_URL
    -> Backend retourne answer + analysis

Configuration photo : PHOTO_PROCESSING_API_URL = URL complete du endpoint d'analyse (ex. http://microservice_photo:8000/analyze). Appel interne MinIO via minio:9000 ; URLs navigateur via MINIO_PUBLIC_ENDPOINT.

Redis limite les rafales de requetes. Les limites sont adaptees selon la route : globale, login et chat.

Cache

Cache TTL Usage
Chat IA CHAT_CACHE_TTL_SECONDS Eviter de rappeler Mistral pour une requete identique.
MiniIO MINIO_IMAGE_CACHE_TTL_SECONDS, maximum 180 s Eviter de regenerer souvent les URLs presignees.

Airflow stale AIRFLOW_STALE_CACHE_TTL_SECONDS Continuer a afficher la page admin si Airflow tombe.

8. Fallback et continuite de service

Les appels externes passent par un client resilient avec timeout, retries et circuit breaker. Le but est d'eviter qu'une dependance externe bloque tout le backend.

Dependance Comportement en panne
Mistral Retour d'une reponse de secours si le fournisseur est indisponible.
Airflow Retour du cache frais, puis stale, puis etat degrade.
Redis Fail-open pour le rate limiting ; cache ignore si Redis tombe.
MiniIO Erreur controlee seulement sur les routes d'image.
microservice_ia /api/chat/recommendations retourne 503 ou 502.
microservice photo /api/chat/images/analyze retourne 503 ou 502.

9. Securite

  • Authentification par JWT Bearer.
  • Hash des mots de passe avec passlib/bcrypt.
  • Routes admin protegees par require_admin.
  • 2FA TOTP disponible pour renforcer la connexion.
  • CORS configure via CORS_ALLOWED_ORIGINS.
  • Headers HTTP de securite via middleware.
  • Controle du proprietaire des images par prefixe users/{id}/chat/.
  • Secrets fournis par .env, jamais par le frontend.

13. Pipeline ML — entrainement et modele

Synthese extraite de microservice_ia/doc/RAPPORT_ML_SYNTHESE.md (pipeline version 2.0). Fichiers : ml/generate_training_data.py, ml/data_prep.py, ml/evaluation.py, ml/train_random_forest.py. Livrables : models/workout_rf_bundle.pkl, models/training_report.json.

Vue d'ensemble du pipeline

1. Generation / modification des donnees (base_profiles.csv → 1500 sessions)
2. Preparation features (12 features, LabelEncoder)
3. Split train / test (80 % / 20 %, stratifie)
4. GridSearchCV + GroupKFold (hyperparametres)
5. Evaluation metriques test hold-out
6. Cross-validation modele final
7. Export .pkl + training_report.json
8. Tests pytest + inference API (/recommandation_exercice, /generate)

13.2 Choix du modele

ModeleClasse sklearnTacheSortie
ClassifierRandomForestClassifierActivite recommandee1 parmi 4 familles
RegressorRandomForestRegressorVolume (series)Entier 2–6

Justification Random Forest : donnees tabulaires mixtes, regles metier non lineaires dans les labels synthetiques, robustesse, peu de preprocessing, aligne avec la spec projet.

Pipeline de recherche : split 80/20 stratifie → GridSearchCV sur train → evaluation test hold-out (300 sessions jamais vues) → CV du modele final → export bundle.

Encodage LabelEncoder : hiit=0, musculation=1, pilates=2, running=3. La matrice de confusion porte sur ces familles, pas sur les noms d'exercices du catalogue API.

13.3 Cross-validation

GroupKFold par user_id (5 folds) : evite la fuite entre les 15 sessions correlees d'un meme utilisateur. Utilise pour GridSearchCV et validation du modele final.

MetriqueTest meanTest stdTrain mean
Accuracy (classification)0,77920,06240,9408

16. Conclusion

Objectifs atteints

  • ✓ Gestion des utilisateurs, authentification et securisation des acces.
  • ✓ Suivi des donnees sante avec stockage SQL et exploitation metier.
  • ✓ Analyse et visualisation des informations utilisateur dans le frontend.
  • ✓ Assistant IA integre au chat avec recommandations personnalisees.
  • ✓ Analyse des images via un microservice photo dedie.
  • ✓ Architecture conteneurisee avec Docker Compose.
  • ✓ Gestion de la charge avec Redis, cache et rate limiting.
  • ✓ Continuite de service grace aux mecanismes de fallback.

HealthAI MSPR repose sur une architecture modulaire qui separe clairement l'interface utilisateur, l'API backend, les donnees, le stockage objet et les traitements IA. Cette organisation permet de faire evoluer chaque partie du projet sans casser l'ensemble de la plateforme.

Le backend FastAPI reste le point d'entree principal : il centralise l'authentification, les regles metier, les acces SQL et NoSQL, la gestion du cache Redis, les fichiers MinIO et les appels aux microservices. Le frontend ne contacte pas directement les services IA ; il passe par les routes securisees du backend, ce qui garde une logique plus simple et plus fiable cote navigateur.

Les deux microservices IA apportent des responsabilites distinctes : microservice_ia fournit les recommandations nutritionnelles et sportives via les routes /api/recommandation_*, tandis que microservice_photo analyse les images envoyees dans le chat. Pour les images, MinIO utilise une URL interne dans Docker pour les microservices et une URL publique uniquement lorsque le navigateur doit afficher le fichier.

Les mecanismes de fallback, le rate limiting Redis, le cache, les tests automatises et la configuration Docker renforcent la continuite de service. Si un service IA est indisponible, l'application peut retourner une reponse degradee au lieu de bloquer completement l'utilisateur.

Cette base technique est donc adaptee a une evolution progressive : amelioration des modeles IA, enrichissement des analyses photo, extension des donnees MongoDB, supervision plus avancee et deploiement plus robuste peuvent etre ajoutes sans remettre en cause l'architecture principale.
